# x-chip-os

Debian (trixie) root filesystems for the NextThing C.H.I.P. (Allwinner R8 /
sun5i), built with [live-build](https://manpages.debian.org/trixie/live-build/lb.1.en.html).

Each flavor is a self-contained live-build project in its own directory.
`headless` is the base; `gui` adds an XFCE desktop; `pocketchip` will follow.
Shared config (kernel apt repo + key, the boot.scr/user hooks, hostname/hosts/
adduser.conf) is **duplicated** into each flavor dir rather than symlinked, so
every flavor is a standalone live-build tree (live-build copies includes with
`rsync -l`, which would turn symlinked includes into dangling links in the
image).

## Layout

```
Makefile            # `make` -> build a flavor's rootfs in a container
Dockerfile          # armv7 build container (live-build)
build.sh            # runs init.sh + lb build + repacks the rootfs tar (in-container)
init.sh             # the `lb config` invocation -- the project's settings
headless/           # the headless flavor (a normal live-build tree)
  config/
    archives/       # the CHIP kernel apt repo (kernel comes from here)
    package-lists/  # what gets installed
    includes.chroot_before_packages/   # flat files dropped in before packages
    includes.chroot_after_packages/    # flat files dropped in after packages
    hooks/normal/   # the things that need code (run flash-kernel, user, greetd)
```

Project settings live in `init.sh` (not hand-edited config files). To change
them, edit `init.sh` and re-run it inside the flavor dir
(`cd headless && bash ../init.sh`). It regenerates `config/*` but leaves your
archives, package-lists, includes, and hooks untouched.

## Building

```
make                 # builds headless -> ./headless-rootfs.tar.gz
make FLAVOR=gui      # builds gui      -> ./gui-rootfs.tar.gz
```

`make` builds the armv7 container and runs `lb build` inside it, so the armhf
rootfs is bootstrapped natively (the host just needs binfmt/qemu registered to
run the arm/v7 image). The finished rootfs is repacked to
`./<flavor>-rootfs.tar.gz` at the repo root, ready for `x-chip-tools` to flash.

## Kernel

The kernel is **not** bundled here -- it's pulled from the CHIP apt repo at
`https://nextthingco.github.io/x-chip-deb-repo/trixie`, configured in
`headless/config/archives/chip.list.{chroot,binary}` (so it installs at build
time and the device can `apt upgrade` it later). The matching signing key ships
alongside as `chip.key.{chroot,binary}`, so live-build registers it and apt
verifies signatures. `init.sh` sets `--linux-packages none` so live-build
doesn't pull a stock Debian kernel; the package list installs `linux-image-chip`
from the CHIP repo instead.

That repo is generated by `x-chip-deb-repo`, which is poked by `x-chip-linux-deb`
whenever new kernel `.debs` are built.

## gui flavor

XFCE on Xorg with the open Mesa/`lima` GPU stack (the R8's Mali-400 is GLES2-only
via lima -- no desktop GL or Vulkan). Login is `greetd` + the `tuigreet` TUI
greeter on vt1, which runs `startx` on login; with no `~/.xinitrc`, that goes
through `/etc/X11/Xsession` and launches `xfce4-session`. The `0300-greetd` hook
points greetd at tuigreet and sets the boot target to `graphical.target`.

Requires the **armmp-based kernel** (full DRM/lima driver set) -- x-chip-linux-deb
now bases the `chip` flavour on Debian's armmp config, so `linux-image-chip` from
the apt repo carries `DRM_SUN4I`/`DRM_LIMA`. (Wiring up the actual PocketCHIP LCD
panel/backlight in the device tree is still a separate, later step.)

## Notes

- No `live-boot`/`live-config`: this image boots from NAND as a normal
  installed Debian (ubifs root), not as a live system.
- The board has no RTC, so `fake-hwclock` is included.
- u-boot and the NAND flashing tooling live in `x-chip-uboot` / `x-chip-tools`.
